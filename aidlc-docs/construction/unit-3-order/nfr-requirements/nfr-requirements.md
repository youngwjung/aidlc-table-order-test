# 비기능 요구사항 - Unit 3: Customer Order Flow

> Unit 1 Foundation NFR을 상속하며, Unit 3 특화 요구사항을 추가합니다.

---

## 1. 성능 (Performance)

| 항목 | 목표 | 비고 |
|---|---|---|
| 주문 생성 API | 500ms 이내 | 메뉴 검증 + DB 쓰기 포함 |
| 주문 내역 조회 API | 300ms 이내 | 읽기 전용, session_id 인덱스 활용 |
| 장바구니 localStorage 동기화 | 즉시 (동기) | JSON 직렬화/역직렬화 |
| 장바구니 합계 재계산 | 즉시 (동기) | 클라이언트 사이드 연산 |
| 주문 확인 모달 표시 | 100ms 이내 | 이미 메모리에 있는 데이터 활용 |

### 클라이언트 성능 고려사항
- 장바구니 상태 변경 시 불필요한 리렌더링 최소화 (`useReducer` 활용)
- localStorage 동기화는 `useEffect`로 비동기 처리 (렌더링 차단 없음)
- 주문 성공 페이지 카운트다운은 `setInterval`로 구현 (cleanup 필수)

---

## 2. 확장성 (Scalability)

| 항목 | 목표 |
|---|---|
| 동시 주문 제출 | 10~30건 (Unit 1 상속) |
| localStorage 장바구니 | 제한 없음 (프로토타입) |
| 세션당 주문 수 | 제한 없음 |

### Unit 3 확장 고려사항
- 주문번호 생성 시 동시성 이슈: 단일 서버이므로 DB 쿼리 + INSERT로 처리 (retry 3회)
- localStorage는 클라이언트 로컬이므로 서버 확장과 무관

---

## 3. 신뢰성 (Reliability)

| 항목 | 구현 |
|---|---|
| 장바구니 데이터 보존 | 주문 실패 시 장바구니 유지 |
| localStorage 복원 | 손상된 JSON 시 빈 장바구니로 초기화 |
| 주문 제출 타임아웃 | 10초 |
| 중복 주문 방지 | 클라이언트 사이드 버튼 비활성화 |
| 에러 복구 | 재시도 가능 (장바구니 보존) |

---

## 4. 보안 (Security)

| 항목 | 구현 |
|---|---|
| 주문 생성 인증 | Table JWT 필수 |
| 멀티테넌트 격리 | store_id 검증 (JWT → URL 일치 확인) |
| 메뉴 가격 조작 방지 | 서버에서 현재 가격으로 재계산 |
| localStorage 보안 | 민감 데이터 미포함 (장바구니만) |
| SQL Injection | SQLAlchemy ORM (Unit 1 상속) |

---

## 5. 사용성 (Usability)

| 항목 | 구현 |
|---|---|
| 장바구니 터치 타겟 | +/- 버튼 최소 44x44px |
| 수량 변경 피드백 | 즉시 반영 (로딩 없음) |
| 주문 제출 피드백 | 로딩 스피너 + 버튼 비활성화 |
| 주문 성공 피드백 | 성공 아이콘 + 주문번호 + 카운트다운 |
| 에러 피드백 | Toast 메시지 |
| 빈 장바구니 | "장바구니가 비어있습니다" 안내 + 비활성화 버튼 |
| 금액 표시 | 천 단위 쉼표 (25,000원) |

---

## 6. 데이터 관리 (Data Management)

| 항목 | 구현 |
|---|---|
| OrderItem 스냅샷 | 주문 시점 메뉴명, 단가 저장 (메뉴 변경에 독립적) |
| 세션 격리 | session_id 기반 주문 조회 |
| Seed 데이터 | 테스트용 메뉴/카테고리 데이터 (Unit 2 대체) |
| 인덱스 | orders: (store_id, table_id, session_id) 복합 인덱스 권장 |

---

## 7. 테스트 요구사항

| 항목 | 범위 |
|---|---|
| 장바구니 로직 | 추가/수정/삭제/초기화/복원 단위 테스트 |
| 주문 API | 생성/조회 통합 테스트 |
| 메뉴 검증 | 유효/무효 메뉴 경계값 테스트 |
| 세션 시작 | 첫 주문 시 자동 시작 테스트 |
| 에러 시나리오 | 빈 장바구니, 무효 메뉴, 네트워크 실패 |
