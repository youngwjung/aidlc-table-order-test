# 비즈니스 규칙 - Unit 1: Foundation

---

## 1. 매장 초기 설정 규칙

### BR-S01: 최초 접속 시 초기 설정
- 시스템에 매장이 하나도 없으면 초기 설정 화면으로 리다이렉트
- 초기 설정 화면에서 매장 정보 + 첫 번째 관리자 계정을 함께 생성
- 초기 설정은 1회만 가능 (매장이 존재하면 초기 설정 불가)

### BR-S02: 매장 식별자 규칙
- 영문 소문자 + 숫자 + 하이픈(-) 만 허용
- 3~30자 제한
- 시스템 전체에서 고유 (UNIQUE)
- 생성 후 변경 불가

### BR-S03: 매장명 규칙
- 1~100자 제한
- 공백 허용, 빈 문자열 불허

## 2. 관리자 인증 규칙

### BR-A01: 관리자 로그인
- 매장 식별자 + 사용자명 + 비밀번호 3개 필드 필수
- 매장 식별자로 먼저 매장 조회 → 매장 내에서 사용자명 조회 → 비밀번호 검증
- 인증 실패 시 구체적 실패 사유(매장 없음/사용자 없음/비밀번호 틀림)를 구분하지 않고 일반 에러 메시지 표시 ("매장 식별자, 사용자명 또는 비밀번호가 올바르지 않습니다")

### BR-A02: JWT 토큰 발급
- 토큰 페이로드: `{sub: admin_user_id, store_id, role: "admin", exp}`
- 만료 시간: 16시간
- 알고리즘: HS256
- 시크릿 키: 환경 변수로 관리

### BR-A03: 관리자 비밀번호 규칙
- 최소 4자 이상 (프로토타입 수준)
- bcrypt 해싱 저장 (salt rounds: 12)
- 빈 비밀번호 불허

### BR-A04: 관리자 사용자명 규칙
- 영문 소문자 + 숫자 + 언더스코어(_) 허용
- 2~30자 제한
- 매장 내 고유

### BR-A05: 로그인 시도 제한
- **기준**: 매장 식별자 + 사용자명 조합별
- **제한**: 3회 연속 실패 시 1분 잠금
- **잠금 해제**: 1분 경과 후 자동 해제
- **성공 시**: 실패 카운터 초기화
- **저장**: 인메모리 (서버 재시작 시 리셋)

### BR-A06: 세션 유지
- JWT 토큰을 localStorage에 저장
- 모든 API 요청 헤더에 `Authorization: Bearer {token}` 포함
- 토큰 만료 시 401 응답 → 로그인 페이지 리다이렉트
- 브라우저 새로고침 시 localStorage에서 토큰 복원

## 3. 테이블 인증 규칙

### BR-T01: 테이블 초기 설정 (관리자 수행)
- 매장 식별자 + 테이블 번호 + 비밀번호 입력
- 설정 정보를 태블릿 localStorage에 저장
- 설정 후 자동 로그인 수행

### BR-T02: 테이블 자동 로그인
- localStorage에 저장된 정보로 자동 인증
- 인증 성공: 테이블용 JWT 발급 → 메뉴 화면으로 이동
- 인증 실패: "관리자에게 테이블 설정을 요청해 주세요" 메시지 표시

### BR-T03: 테이블 JWT 토큰
- 토큰 페이로드: `{sub: table_id, store_id, table_number, role: "table", exp}`
- 만료 시간: 16시간
- 테이블 비밀번호: 평문 저장 (내부 인증용)

### BR-T04: 테이블 번호 규칙
- 양의 정수 (1 이상)
- 매장 내 고유
- 테이블 비밀번호: 1~20자

## 4. 테이블 세션 규칙

### BR-TS01: 세션 시작
- 테이블에 세션이 없는 상태(session_id = NULL)에서 첫 주문 생성 시 자동 시작
- 세션 시작 시: UUID v4로 session_id 생성, session_started_at 기록
- 이미 활성 세션이 있으면 기존 세션 유지

### BR-TS02: 세션 종료 (이용 완료)
- 관리자만 수행 가능
- 해당 세션의 모든 주문을 OrderHistory로 이동
- 테이블의 session_id = NULL, session_started_at = NULL로 리셋
- 이후 새 고객의 첫 주문 시 새 세션 자동 시작

### BR-TS03: 세션 기반 데이터 격리
- 고객은 현재 테이블의 활성 session_id에 해당하는 주문만 조회 가능
- 세션 종료 후에는 해당 세션 주문이 이력으로 이동되어 고객 화면에 표시되지 않음

## 5. 주문 번호 규칙

### BR-ON01: 매장별 일일 순번
- 형식: 3자리 숫자 문자열 (예: "001", "002", "100")
- 매장별 독립 순번
- 매일 자정(00:00) 기준 "001"부터 재시작
- 생성 방법: 해당 매장의 당일 주문 수 + 1

## 6. 멀티테넌트 규칙

### BR-MT01: 데이터 격리
- 모든 API 엔드포인트에서 JWT의 store_id 기준 데이터 접근
- 다른 매장의 데이터에 접근 불가
- URL 경로의 store_id와 JWT의 store_id 일치 검증

### BR-MT02: 관리자 접근 범위
- 관리자는 자신이 속한 매장의 데이터만 접근 가능
- 매장 간 데이터 교차 접근 차단

## 7. 공통 검증 규칙

### BR-V01: 가격 검증
- 0 이상의 정수 (원 단위, 소수점 없음)
- 최대값: 10,000,000 (천만 원)

### BR-V02: 수량 검증
- 1 이상의 양의 정수
- 최대값: 99

### BR-V03: 총 금액 계산
- total_amount = Σ(unit_price × quantity) for all OrderItems
- 주문 항목 변경 시 자동 재계산
